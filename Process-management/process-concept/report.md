# Report(1): Process Concept & Observation

## 1. 목표
프로그램과 프로세스의 차이를 이해하고, 실행 중 프로세스의 메모리 구조와 상태를 직접 관찰

## 2. 이론 요약
- 프로세스는 실행 중인 프로그램이다.
- 프로세스는 code, data, heap, stack으로 구성된 주소 공간을 가진다.
- 프로세스는 new, ready, running, waiting, terminated 상태를 오간다.
- OS는 PCB를 통해 프로세스 상태를 관리한다.
- context switch는 필요한 동작이지만 오버헤드가 발생한다.

## 3. 실습 내용
### 3-1. ps / /proc 관찰
실행한 명령어:
- `ps -o pid,ppid,stat,ni,pri,comm -p $$`
- `cat /proc/self/status | head -30`
- `cat /proc/self/maps | head -20`

관찰한 내용:
- 현재 셸도 하나의 프로세스였다.
- PID와 PPID를 확인할 수 있었다.
- /proc을 통해 상태와 주소 공간 일부를 확인할 수 있었다.

### 3-2. mem_layout.c 실행
출력 결과에서 함수, 전역 변수, 지역 변수, malloc 메모리의 주소가 '서로 다른 영역'에 배치되는 것을 확인했다.

## 4. 해석
- 프로그램 파일 자체는 정적이지만, 실행되면 프로세스가 되어 상태를 가진다.
- 지역 변수는 스택에, 동적 할당 메모리는 힙에 존재한다.
- OS가 프로세스를 다시 실행하려면 PC, 레지스터, 메모리 상태 등의 정보가 필요하다.
- 따라서 PCB와 context switch 개념이 필수적이다.

## 5. 실무 연결
- PID/PPID는 서버 프로세스 추적에 중요하다.
- ps, /proc 같은 도구는 실제 운영 환경에서도 자주 사용된다.
- 프로세스 관찰은 장애 분석과 성능 분석의 출발점이다.

## 6. 실수와 수정 과정
초기 실습에서는 `mem_layout.c`에

printf("\nPress Enter to exit...\n");
getchar();
를 추가했음에도 불구하고, 실행 중인 mem_layout 프로세스를 /proc/<PID>로 제대로 관찰하지 못했다.

원인을 확인한 결과, return 0;이 위 두 줄보다 앞에 위치해 있었기 때문에 프로그램이 getchar()에 도달하기 전에 이미 종료되고 있었다. 즉, 입력 대기 상태로 남아 있어야 할 프로세스가 실제로는 종료된 상태였고, 그 결과 잘못된 PID를 관찰하게 되었다.

수정 후에는 getchar()가 return 0;보다 앞에서 정상적으로 실행되도록 고쳤고, 그 결과 mem_layout 프로세스가 S (sleeping) 상태로 유지되는 동안 /proc/<PID>/status와 /proc/<PID>/maps를 통해 정확한 프로세스 정보를 관찰할 수 있었다.

이 과정을 통해 프로세스 관찰에서는 코드 내용뿐 아니라 제어 흐름과 종료 시점이 매우 중요하다는 점을 확인했다.

## 8. 궁금한 점
- make ./mem_layout 실행 시 프로그램이 멈추게 됨 -> 이 녀석(멈춘 놈)은 어떻게 처리 되는건가?
Answer: 프로그램이 getchar()같은 입력 대기 함수에 멈춰 있다면, 그 프로세스는 보통 입력을 기다리는 blocked/sleeping 상태로 돌입 -> CPU를 계속 먹으면서 "바쁘게 기다리는 것"이 아님.

### mem_layout.c flow chart
- 실행 중이던 프로세스가 getchar() 호출
- stdin 입력이 아직 없음
- OS가 이 프로세스를 입력 이벤트 대기 상태로 둠
- 사용자가 Enter 입력
- 프로세스가 다시 ready/running으로 돌아옴
- 남은 코드 실행 후 종료
